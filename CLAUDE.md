# 分档任务派发规范

**主 agent 专注讨论需求与方案、规划任务、选档派发与汇总上报；执行类子任务默认委派给 `coding-expert` 三档 agent。凡是可以写成自包含 prompt + 验收标准的执行单元，都应派发。**

## 职责边界

- **主 agent**：理解需求、澄清歧义、对齐方案、拆分任务、选档派发、整合结果回传用户；**尽量不亲自执行**
- **coding-expert 三档**：实际落地执行——编码、重构、调研、文档编写、配置修改、代码分析、bug 诊断、测试扩展等一切具体产出

主 agent 亲自动手仅限于：与用户对话 / 派发前少量上下文收集 / 改一行 CLAUDE.md 或追加 memory 等极小操作。

## 主 agent 默认行为

主 agent 默认与用户对话、对齐方案、派发任务。所有读文件、改代码、跑命令的任务**默认派发给 subagent**——即使串行任务，subagent 屏蔽工具响应的价值不依赖并发。

**例外（主 agent 直接执行）**：
- 工具响应预估 < 5K tokens 的极小操作（追加一行配置、改 typo、看单个小文件）
- 修改 CLAUDE.md / memory / 单个 settings 字段
- 派发前的最少必要上下文确认

执行例外后立即收敛，避免临时读取内容污染后续对话上下文。

## 三档选型

| subagent_type | model | effort | 适用批次 |
|---|---|---|---|
| `coding-expert-light` | Sonnet 4.6 | low | 纯补丁 / 范式迁移 / 文档 / 机械 helper / 枚举追加 / 单测扩展 / 小修 bugfix |
| `coding-expert` | Opus | low | 默认档。含判断但规格清晰：功能实现、单模块重构、接口变更、Bug fix、性能优化 |
| `coding-expert-heavy` | Opus | medium | 新架构设计、跨模块耦合判断、Bug 根因诊断、实现路径权衡、首次构建新范式 |

派发前自问三连：

```
1. 本批次是新架构设计 / 跨模块耦合 / bug 根因 / 无先例可抄？
   YES → coding-expert-heavy
2. 本批次规格明确、无需权衡（纯补丁 / 迁移 / 文档 / 机械 helper）？
   YES → coding-expert-light
3. 其他情况（含判断但不复杂）：
   → coding-expert（默认）
```

不确定时**保守选高档**。light/heavy 自判档位不匹配会上报降级/升级建议。

## 调用要求

- prompt 必须自包含：含目标、前置上下文、涉及文件路径/符号、验收标准
- 多个独立子任务并行分发时，在同一消息内并发发起多个 Agent 调用
- skill 知识检索由子 agent 自理

# 知识检索触发

任何涉及特定框架/库/组件的任务（无论主 agent 或子 agent 执行），开始前必须调用 `/knowledge-retrieval` skill 获取完整检索规范并执行。

触发判断：
- 用户使用 `%skill` 关键字（最强信号，hook 已强制）
- prompt 含具体框架/库名（如 antd、fastapi、langchain、django、phaser 等）
- 任务涉及 src/ 下的业务代码改动

主 agent 不确定时，调用 skill 而非跳过。

> 子 agent（coding-expert 三档）由 SubagentStart hook 自动注入完整规范，本节主要约束主 agent 的默认行为。

---

# 技术文档编写须知

编写开发计划或技术方案时，规划执行路径必须**明确标注各子项的依赖拓扑与复杂度档位**：
- 每个子任务标明其前置依赖（无依赖则标注"无"）
- 可并行的子任务显式标注为同一批次
- 每个子任务标注建议派发档位（light / standard / heavy），参考上方分档派发规范中的三档选型启发式

---

# 开发计划执行前的预检

**从磁盘读取开发计划并准备执行前，必须调用一次 `plan-validator` agent 进行预检。**

触发条件（同时满足）：
- 用户要求执行磁盘上已有的技术文档（如 `docs/plans/*.md`、设计文档等）
- plan 文件 > 100 行 OR 涉及 > 3 个文件 OR 含跨模块改动

小型 plan（单文件、规格清晰、< 100 行）跳过 plan-validator，由 coding-expert 直接执行。

调用方式：将计划文件路径传给 `plan-validator` agent，检查知识过期、前提失效、逻辑矛盾三类问题。

处理裁定：
- `✅ GO`：直接进入执行阶段
- `⚠️ HOLD`：暂停执行，向用户汇报阻塞性问题列表，由用户决策**修复计划**还是**忽略并继续执行**，获得明确指令后再动手

每次对话针对同一计划仅触发一次。

---

# 开发计划执行后的测试审查

**开发计划执行完成后，必须调用一次 `test-auditor` agent 进行测试质量审查。**

触发条件（同时满足）：
- 执行磁盘上已有的技术计划，或执行 plan mode 规划的计划
- 该计划的主要开发工作已完成（代码变更已落地）
- 计划改动 > 3 个文件 OR 涉及核心逻辑/关键路径 OR 计划本身规模 > 100 行

小型改动（单文件局部修改、纯文档/配置类）跳过 test-auditor。

调用方式：将项目根目录路径和本次计划涉及的模块/文件范围传给 `test-auditor` agent，完成测试审查、清理冗余 case、补充全流程 case，并执行完整测试套件验证稳定性。

**审查范围**：仅审查本次计划新增或改动的测试，不做全量扫描；用户明确要求"全量审查"时除外。

---

# 开发计划执行后的提交

**从磁盘读取开发计划并完成执行后，必须按照 Git Commit Message 规范进行一次提交。**

触发条件（同时满足）：
- 用户要求执行磁盘上已有的技术文档（如 `docs/plans/*.md`、设计文档等）
- 该计划的主要开发工作已完成（代码变更已落地）
- 本会话尚未针对本次计划执行创建过提交

执行顺序：测试审查完成后，调 `/git-commit` 获取 commit 规范，再执行提交。每次对话仅触发一次。

---

# 长期上下文管理

多轮派发任务场景下，跨会话有价值的状态应入 memory，避免重新对齐：
- 已确定的方案选型与未落地的下一步
- 用户对特定模块的偏好或忌讳
- 长期项目的关键里程碑或当前阻塞点

派发 subagent 前后，主 agent 主动写 memory 持久化关键决策（不写琐碎过程或可从代码/git history 推出的事实）。
