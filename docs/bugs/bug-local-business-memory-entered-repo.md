# Bug: 本地业务记忆误入配置仓库

## 现象

`memory.md` 中包含只应保存在本机的内部业务项目、内部工具链和构建排障记忆。
这些内容随配置仓库提交并推送，导致本仓历史中也保留了不应共享的业务上下文。

## 根因 (6 要素)

1. **触发条件**：排障后把长期记忆直接追加到本仓 `memory.md`，没有区分“通用配置
   经验”和“仅本机可见的内部业务经验”。
2. **期望链路**：本仓 `memory.md` 只保存可共享、低敏、跨项目复用的配置和工具经验；
   内部业务上下文写入本机私有记忆文件。
3. **实际链路**：本仓记忆文件承担了全局记忆和本机私有记忆两种职责，提交时缺少
   敏感关键词和业务上下文检查。
4. **关键假设失效**：误以为“个人配置仓库”可以承载所有个人记忆；实际该仓库会同步
   到远端并被其他安装点复用。
5. **旁证**：历史提交中已经出现多次业务构建和内部工具排障内容；仅改当前 HEAD
   不能清除旧 commit 中的记录。
6. **实现偏差**：没有为本机私有记忆建立独立落点，也没有在提交前把内部业务关键词
   从仓库文件和历史中隔离。

## 修复原则

- 把内部业务记忆迁移到本机 `~/.agents/memory.md`。
- 从本仓 `memory.md` 删除对应内容。
- 重写仓库历史，确保旧 commit 的 `memory.md` 中也不再包含这些业务记忆。
- 仓库内只记录通用原则，不记录具体内部项目、内部工具链或业务构建细节。
