# 全局记忆

已知陷阱、过往 bug 记录于当前宿主的 memory 文件。**两个强制查阅时机**：

1. **动手前**——开工任何 coding / 调试 / 排查任务前，先扫一遍相关条目。
2. **遇到报错 / 测试失败 / 异常行为时**——在做新的根因分析或修复尝试之前：
   - 命中已记录的坑 → 直接按记录中的解法走
   - 未命中 → 启动 bug-analysis 流程，分析定型后回写到 memory.md

跳过此查阅视为流程违规，回退重做。

> **原因**：踩过的坑不应该再踩第二遍；遇到报错时优先比对历史记录可以省去重新摸索的成本。命中时直接走旧解法是最大 ROI 路径；未命中时回写形成正反馈循环，下次就能直接命中。设成"流程违规"而非"建议"是因为软建议会被 rationalize 跳过。

# Bugfix 流程约束

测试发现 / 人类提交的 bug，禁止直接进入修复。必须先完成根因分析，输出
`bug-analysis.md`，再由用户确认后执行修复。

> **原因**：测试失败 / 报错的表象 ≠ 根因。直接动手修复经常只是打症状（如把断言改宽、加 try-catch 吞异常），留下隐患甚至引入新 bug。强制产出结构化分析文档迫使先把调用链、影响范围梳理清楚；用户确认环节避免 agent 在错误假设上推进。

## 根因分析必须回答（缺任一项不得修复）

**现象**：测试在哪里失败，错误信息是什么
**调用链**：从测试入口到失败点的完整业务逻辑链（非堆栈）
**根因假设**：列出 1-3 个候选假设
**验证方式**：日志 / 单测 / 断点
**根因确认**：一句话结论
**影响范围**：根因还可能在哪些其他路径上触发

> **原因**（逐项）：
> - **现象**：精准定位失败点，避免"测试挂了"这种模糊描述。
> - **调用链强调业务逻辑链而非堆栈**：堆栈只反映函数调用，业务逻辑链才能暴露"为什么会走到这一步"，这是根因思考的基础。
> - **1-3 个候选假设**：强制至少列 1 个上限 3 个，逼出多种可能避免单一直觉拍板，也防止候选爆炸失焦。
> - **验证方式**：避免假设停留在猜测——必须落到可操作的验证手段。
> - **根因确认收敛成一句话**：未收敛即未理解，不得进入修复。
> - **影响范围**：bug 几乎总有多发路径，只修一处会留隐患。

## 修复方案要求

- 必须针对根因，不得只针对症状
- 说明为什么此修复不会在其他路径引入新问题
- 补充覆盖根因路径的单测（非复现原失败用例）

> **原因**：
> - 症状修复治标不治本，下次换个触发路径还会复发。
> - 修复经常会副作用波及其他路径，强制 agent 自己论证一遍能逼出回归思考。
> - 单测必须覆盖"根因路径"，而不是仅让"原失败用例"再过——后者是给测试镀金，前者是给代码上保险。

## 修复后验证

- 原失败用例必须通过
- 影响范围中列出的路径必须有对应测试覆盖并通过
- 全量测试重跑，不得有新增失败
- 新增失败视为引入回归，回退本次修复重新分析

> **原因**：原用例通过只能证明症状消除；影响范围有测试才能证明根因覆盖；全量重跑是最后一道防线防修复引入回归。回退而非修补是因为"边修边引入"会陷入打地鼠循环。

## 修复后异源复审（external-llm-review）

bugfix 流程硬约束，优先级高于 §5 的"高风险任务才上"。

> **原因**：bug 修复放过盲点的代价远高于 feature 开发——bug 已经在用户/测试环境暴露过一次，再放过同款问题代价指数级上升。同族模型对自身修复倾向 normalize 通过，必须异源抓盲点。所以 bugfix 默认强制外源，不走 §5 的"按风险决定"。

### 触发与范围

- **何时跑**：修复后验证全部 ✅ 之后；fix 未收敛前不跑
- **审查范围**：`BASE..HEAD` 严格对齐本次修复 commit
- **必须输入**：`bug-analysis.md`（根因结论 + 影响范围）+ 本次 diff

> **原因**：fix 未收敛跑外源 = 烧 token 看噪声；范围严格对齐避免无关变更稀释 reviewer 注意力；bug-analysis.md 是 reviewer 判断"修复是否真覆盖根因"的必要上下文，没有它外源 reviewer 只能做泛化 code review。

### 审查重点（dispatch prompt 必须明确要求 reviewer 回答）

1. 修复是否真正针对 `bug-analysis.md` 确认的根因？是否仍在打症状？
2. `影响范围` 中列出的其他路径，本次修复是否触达？有无遗漏？
3. 修复是否在其他路径引入新失败模式（边界、并发、错误处理）？
4. 新增单测是否覆盖根因路径？

> **原因**：不约束 prompt 时外源会自动滑向泛化 code review（命名、风格、cyclomatic complexity 等），与 bugfix 的核心关切错位。4 问强制 reviewer 围绕"根因覆盖 + 影响范围 + 新失败模式 + 测试质量"作答，这是 bugfix 评审唯一关心的四件事。

### 结果处置

- Critical / Important → 回到根因分析阶段，更新 `bug-analysis.md` 或修复方案后重走流程
- Minor → 用户决策是否本次处理
- ✅ 通过 → 可提交
- 综合判断 4 步规则与协议切换详见 `external-llm-review` skill

> **原因**：Critical/Important 强制回退避免 agent 用"权衡考虑"为绕过开脱；Minor 留用户决策避免被噪声拖死；4 步规则用于消化 LLM 误报。

### 豁免条款（须在提交说明中标注豁免原因）

- 修复 diff < 10 行且仅涉及单个无外部依赖的函数
- 纯文档 / 配置 / 拼写类 bug
- 项目合规策略不允许源码 diff 出域（须在 `bug-analysis.md` 显式记录）
- 未配 `.env` 或外部凭据时优先补齐凭据，不默认走豁免

> **原因**：< 10 行单函数无外部依赖时，外源审查的边际价值低于 token 成本；纯文档/配置无逻辑可审；合规禁止时硬性豁免但必须留痕；凭据缺失不算豁免理由（应该补齐而非绕过），明确写出来避免被作为常态豁免借口。

# Superpowers 流程增强

## 1. 反幻觉：每阶段强制知识检索 + Web 补充调研

- `brainstorming` 与 `writing-plans` 阶段必须调用 `knowledge-retrieval`
  （前者侧重选型与架构，后者侧重落地模式与 task 拆分）
- 每个 implementer subagent 动手前必须自己跑 `knowledge-retrieval`；
  约束写入 dispatch prompt，主 agent 不代办
- **Web 补充调研**：`knowledge-retrieval` 之后，命中以下任一情形必须再用
  当前宿主提供的 Web 搜索 / 抓取工具拉最新资料：
  - 涉及第三方框架 / SDK / API 的版本号、breaking changes、deprecation
  - < 1 年内发布或仍在活跃迭代的协议、库、规范
  - 安全相关（CVE、token 格式、auth flow、加密协议变更）
  - 本地 skill 与已读文档均未覆盖的外部 API
- Web 结果与本地 skill 冲突时，当前任务以官方最新文档为准；**不得自动
  回写 skill 库**——先列出"本地 vs 最新"差异请用户确认，同意后再回写

> **原因**：
> - LLM 知识截止 + 训练数据中的代码常含过期 API，凭直觉写最容易翻车。强制先检索 = 用本地高质量 skill 库压制幻觉。
> - subagent 上下文独立，主 agent 替它检索的结果无法稳定传递；让 subagent 自己检索更可靠。
> - Web 补充覆盖训练截止后的窗口期：版本/CVE/新协议这几类变化最快，必须实时查。
> - 自动回写 skill 库的风险是把过期信息覆盖正确信息（agent 无法可靠判断本地与最新哪个对），用户审核环节强制人为校验。

## 2. 可并发：DAG 拓扑而非串行

- `writing-plans` 必须对子任务做 DAG 依赖分析，明确执行顺序与并发集合
- `subagent-driven-development` 按 DAG 编排：独立 task 并行派发
- 并发 task 在独立 git worktree 中隔离：
  - 主 agent 留在协调层，**不调用 `using-git-worktrees` skill**，直接
    `git worktree add` 一次性建好所有 worktree；须履行以下安全契约：
    - **目录优先级**：`.worktrees/`（已存在则复用）> `worktrees/` > 默认新建 `.worktrees/`
    - **`.gitignore` 校验**：首次 add 前一次性确保 worktree 目录已被忽略，
      未忽略则 add+commit；后续并发 add 不重复校验
    - **Submodule guard**：若 cwd 在子模块内，先 `cd` 到 superproject root 再建
    - **Sandbox 降级**：`git worktree add` 因权限拒绝失败时整批回退到
      串行执行，并提示用户
  - 路径写入各 subagent prompt；subagent 在已就绪的 worktree 内执行任务
  - 并发结束后合并工作树；自动合并失败的冲突提请用户决策

> **原因**：
> - 串行浪费独立任务的并行潜力；DAG 显式声明依赖才能安全并发。
> - `using-git-worktrees` skill 是 "把当前 agent 移入 worktree" 的语义，主 agent 调用会脱离协调位；直接用 `git worktree add` 保持主 agent 在协调层。
> - 目录优先级：`.worktrees/` 是约定俗成的隐藏目录；已存在时复用避免目录爆炸。
> - `.gitignore` 校验：worktree 目录未忽略会污染 working tree；一次性校验避免并发 add 时的竞态冲突。
> - submodule guard：在子模块内执行 `worktree add` 会建到子模块的独立 .git 里，与 superproject 的协调不兼容。
> - sandbox 降级：权限受限的环境下不应硬卡，降级到串行至少能跑完。

## 3. subagents worker 执行策略

- 使用 `superpowers:subagent-driven-development` / `dispatching-parallel-agents`
  时，implementer / worker 默认走 `opencode-deepseek-worker` skill，由
  `opencode + deepseekv4pro` 在隔离 worktree 中执行实际任务并产出候选 diff。
- reviewer / final reviewer 不走 DeepSeek worker；仍由当前宿主按 review skill
  与外源复审规则执行。
- 主 agent 负责拆 task、写清 write scope / validation、审查 worker diff、合并
  approved 变更并在主 checkout 重跑验证；不得把架构决策或最终判断交给 worker。

> **原因**：旧的统一模型分档入口不再维护；当前可复用价值集中在 OpenCode DeepSeek worker 的执行协议。把 implementer / worker 默认交给 `opencode-deepseek-worker` 可以让 Superpowers 的实际编码执行有固定产线，同时保留主 agent 的任务拆分、review、合并和验证职责。DeepSeek worker 只产候选 diff，不承担架构和最终判断，避免把执行能力误当成审查能力。

## 4. 不阻塞：subagent 后台执行

- 所有 subagent 统一后台运行，主对话保持响应
- 依赖未满足的 task 须等前驱完成后再派发

> **原因**：前台 subagent 会卡死主对话；后台模式让用户随时可中断/查询。但后台模式不能破坏 DAG——依赖关系是逻辑约束，不因执行模式而消失。

## 5. 完整性：终态校验 + 工作树干净

- 工作流完成后对照 `writing-plans` 产出逐项核实
- 存在未提交变更须完成一次提交

> **原因**：subagent 并发派发后容易遗漏某个 task 的产物；终态对照 plan 是最后一道防线。未提交变更留在工作树里下次会被混淆，必须收尾提交。

## 6. 异源复审：抓同族盲点

### 触发决策

| superpowers 评审 | 是否需外源 |
| - | - |
| spec-compliance reviewer | 否 |
| code-quality reviewer（每个 task） | 是 |
| final code-reviewer（所有 task 完成时） | 是 |

> **原因**：
> - spec-compliance 评的是 plan 一致性，外源不知道 plan 上下文，价值低。
> - code-quality 评的是代码质量，同族模型对自身生成代码倾向 normalize 通过（生成时就用同一组启发式判断了），需异源抓盲点。
> - final code-reviewer 还要考察跨 task 合并面与整体一致性，比单 task 更需要异源视角。

### 调用约束

- 同族 code-quality ✅ Approved 之后才跑外源
- 外源 `BASE..HEAD` 与同族评审区间严格对齐
- 调用方式、协议切换、综合判断 4 步规则见 `external-llm-review` skill
- 仅在高风险任务 / 结论不放心 / 项目策略强制要求时上

> **原因**：
> - fix 未收敛时跑外源 = 烧 token 看一堆同族已经会发现的问题，没意义。
> - 范围对齐保证两边评的是同一份代码，结论才可对比。
> - 不每个 task 都跑：外源调用是真金白银的 API 费用，按风险/价值决定才划算。

### 何时不必用

- 任务纯文档 / 配置（spec 评审、yaml 校验等）
- 模块作用域 < 50 行且无外部依赖
- 未配 `.env` 也未 export 凭据
- 项目合规策略不允许源码 diff 出域

> **原因**：
> - 纯文档配置外源没有专业判断价值。
> - < 50 行无外部依赖时同族判断已足够，盲点空间小。
> - 没凭据是物理不可达，不是规则豁免。
> - 合规出域属于硬法律红线，规则必须让步。

## 7. 决策报告格式约束

### 必填（主报告）

每个决策项总长不超过 5 行，按以下模板：

**[决策项]**：我们需要决定 ___（业务语言，不用技术术语）
**推荐**：___ ，因为 ___（一句话，聚焦最关键理由）
**不选其他的原因**：___ （一句话，聚焦最致命的短板）
**如果选错**：___ 时会暴露，修复代价是 ___（低/中/高）

> **原因**：用户审决策报告的目标是"2 分钟内能拍板"，不是"读完所有调研材料"。5 行限制强制 agent 抽取关键信号；推荐+不选+错选代价三段式给用户全部决策必要信息（选什么、为什么不选别的、错了多严重）。业务语言要求避免技术术语让非技术 stakeholder 也能参与决策。

### 禁止在主报告出现

- 特性对比表（移入附录）
- 技术术语未解释直接使用
- 超过 2 个备选方案的并列描述
- "各有优劣"类结论（必须给出明确推荐）

> **原因**：
> - 特性对比表是详细信息，放主报告会冲淡决策核心。
> - 未解释术语会逼用户脑补，增加误判风险。
> - > 2 个并列方案会触发决策瘫痪。
> - "各有优劣"等于 agent 把决策推回给用户，违背决策报告的本职——必须给出明确推荐及理由。

### 附录（用户主动要求才展开）

完整调研细节、benchmark 数据、备选方案分析

> **原因**：详细资料对深挖必要，但默认隐藏避免污染主决策视野；按需展开符合 progressive disclosure 原则。

## 8. TDD 开发准则

**执行流参照当前环境中的 TDD skill / workflow** 的 RED-GREEN-REFACTOR
完整规范；若存在 `superpowers:test-driven-development`，优先按其流程执行。
coding 任务原则上严格按该流程执行。

**本仓豁免条款**（满足任一可免走 TDD）：

- 单行修改（变量名、配置值、拼写）
- 目标代码已有覆盖该变更路径的测试
- 纯文档 / 配置类变更

冲突时本节豁免优先于 skill 判定（user instructions > skill）。其余情形违反
视为实现无效，回退重做。

> **原因**：
> - 复用 superpowers skill 完整规范避免重写 RED-GREEN-REFACTOR 细节。
> - skill 内置的 "Too simple to test" rationalization 判定太严，对本仓的轻量改动会造成不必要的开销——3 条豁免按"成本 < 测试编写"的实际边界放宽。
> - user instructions > skill 是优先级原则；违反不豁免条款则回退重做避免边写边纵容。

### 分层测试策略

#### 三层定义

| 层级 | 验证对象 | 典型运行时长 | 隔离度 |
| - | - | - | - |
| **单测** | 单个函数/方法的输入输出与边界 | 毫秒级 | 完全隔离 |
| **集成测试** | 单个模块对外契约 | 秒级 | 部分真实（外部依赖 stub） |
| **e2e 测试** | 端到端用户流程 | 十秒～分钟级 | 完全真实 |

**冒烟测试** 单独定位为 CI 入口的健康检查，不与上述三层混用。

> **原因**：
> - skill 全程按 unit test 周期写（秒级 Verify RED、no-mock real code），未覆盖集成/e2e 的差异——必须在本仓补三层定义。
> - 测试金字塔的核心维度是"反馈速度 vs 真实性"权衡：单测快但靠 mock，e2e 真实但慢，集成居中。三层是这个权衡的标准切分。
> - 冒烟在行业里有明确含义（CI 入口的 happy path 健康检查），把"模块级测试"叫成冒烟会造成协作认知混乱——所以必须单独定位。

#### 最小测试覆盖契约

| 开发任务粒度 | 最小必补 | 按影响追加 |
| - | - | - |
| 细粒度（单函数/方法） | 单测 | 落在模块对外契约 → 加集成；落在关键用户路径 → 加 e2e |
| 模块级（多函数协作的对外行为变更） | 单测 + 集成 | 落在关键用户路径 → 加 e2e |
| 需求级（新功能 / 跨模块流程） | 单测 + 集成 + e2e | — |

**关键用户路径**：登录鉴权、付费/订单、核心数据写入、对外 API 契约等
线上故障直接影响业务/用户的路径。不确定时归类为关键路径。

> **原因**：
> - 任务粒度 → 测试层级不是单射映射：一个"改函数"任务如果落在付费路径上，照样要补 e2e。表格表达的是"最小必补"而不是"只补这个"，按影响列就是处理这种例外。
> - 关键用户路径必须明确口径，否则 agent 会按主观感觉判断；不确定时归为关键是 over-cover 偏好，宁愿多写测试也不漏防御。

#### e2e 准入门槛

- e2e 仅覆盖关键用户路径；非关键需求用"集成测试 + 手动验证清单"替代
- e2e 套件总运行时上限 **5 分钟**，超限必须砍 / 拆并行
- 同一路径已有 e2e 覆盖时，新需求优先扩展现有用例断言而非新建
- e2e 失败必须 < 24h 内修复或暂时跳过（带 ticket 编号）

> **原因**：
> - e2e 维护成本是单测的 10 倍以上，失控的 e2e 套件会拖垮迭代速度。经典金字塔：e2e < 10%，单测 > 50%。
> - 5 分钟阈值是经验值：超过这个时间 dev 就会跳过本地跑 e2e，e2e 沦为 CI-only，反馈周期从分钟级退化到 PR 级。
> - 扩展优先于新建避免冗余路径——同一路径多条 e2e 是维护噩梦。
> - 24h 红灯硬约束防"红灯麻木"：长期红灯会导致团队对 e2e 失败麻木，后续真实失败被混在噪声里漏掉。带 ticket 跳过比死着挂 CI 更健康。

#### e2e 的 RED 阶段调整

- 启动 e2e 前先跑环境健康检查（端口 / 浏览器 / seed 数据）
- 失败信息含 `connection refused` / `timeout` / `undefined is not a function`
  等环境特征时，先排查环境再回到 RED 判定
- e2e 的 mock 边界：有副作用的外部依赖（付费、邮件、短信）必须 stub；
  只读类外部 API 按真实成本与稳定性决定

> **原因**：
> - skill 的 Verify RED 假设"失败 = 特性缺失"，这在 unit test 成立，但 e2e 易被环境噪声污染（端口被占、浏览器没起、seed 数据未灌），如果不先做健康检查就盲信失败原因，会误把环境问题当成"特性缺失通过 RED"。
> - 环境特征关键词是常见环境噪声的指纹，命中时强制环境排查避免误判。
> - 有副作用依赖必须 stub：测试不应触发真实付费/邮件/短信，否则产生不可逆副作用且污染真实数据。只读 API 看具体场景决定——稳定且免费的可以直连，不稳定或收费的应 stub。
